# BriefUp 파이프라인 품질 개선 보고서

> 자동화된 콘텐츠 파이프라인의 신뢰성과 품질을 어떻게 높였는가

---

## 배경 — 왜 이 문제를 해결해야 했나

BriefUp은 매일 자동으로 아티클을 수집하고 요약·퀴즈를 생성해 유저에게 제공한다.  
이 파이프라인이 **완전 자동화**된 만큼, 잘못된 내용이 검증 없이 유저에게 전달될 위험이 존재한다.

평가 기준으로 적용한 프레임워크:
- **RAG 평가 지표**: Faithfulness, Context Relevance, Groundedness, Hallucination Rate
- **교육 콘텐츠 품질 프레임워크**: QUEST (Quality·Uniqueness·Effort·Structure·Transparency), ConQuer

---

## 개선 전 구조의 문제점

```
[개선 전]

아티클 수집 (arxiv + Tavily)
    ↓  URL 중복 제거만
    ↓  ← 크로스런 중복 없음, 관련성 판단 불가
GPT-4o-mini → 요약 생성
    ↓  ← 원문 faithfulness 검증 없음
GPT-4o-mini → 퀴즈 생성
    ↓
GPT-4o-mini → 퀴즈 검증  ← 동일 모델 자기검증
    ↓
DB 저장
```

---

## 문제 1 — 동일 모델 자기검증 (Self-Verification)

### 문제
퀴즈 생성과 검증을 **같은 모델(GPT-4o-mini)**이 수행했다.  
같은 모델은 같은 편향과 지식 오류를 공유하므로, 생성 시 틀린 내용을 검증 시에도 통과시킨다.  
이는 사람에게 자기 글을 스스로 교정시키는 것과 동일한 구조적 한계다.

### 해결 방법 — 이종 모델 교차검증 (Cross-Model Verification)
| 단계 | 개선 전 | 개선 후 |
|------|---------|---------|
| 퀴즈 생성 | GPT-4o-mini | **GPT-5** |
| 퀴즈 검증 | GPT-4o-mini (동일) | **Claude Haiku** (다른 모델) |

서로 다른 회사의 서로 다른 아키텍처로 훈련된 두 모델을 사용해,  
한 모델의 blind spot을 다른 모델이 잡아낸다.

### 개선 효과
- 구조적으로 동일 모델 blind spot 해소
- 검증 단계를 통과한 퀴즈가 실제로 두 독립 모델의 동의를 얻은 콘텐츠임을 보장
- **퀴즈 통과율**(`quiz_pass_rate`)을 런마다 측정해 품질 추이 추적 가능

## 문제 2 — 요약 faithfulness 미검증 (Faithfulness)

### 문제
요약 프롬프트에 "원문에 없는 내용 추가 금지"라고 지시했지만, **실제로 지켜지는지 검증하는 코드가 없었다.**  
GPT가 원문을 왜곡하거나 외부 지식을 추가해도 탐지 불가능했다.  
요약 품질이 낮으면 이를 기반으로 생성된 퀴즈도 연쇄적으로 품질이 저하된다.

### 해결 방법 — 요약 후 즉시 교차 faithfulness 검증

```
GPT-5 → 요약 생성
    ↓
Claude Haiku → 원문 ↔ 요약 대조 검증
    • 원문에 없는 주장 감지
    • 수치·사실 왜곡 감지
    • faithfulness 점수 (0.0 ~ 1.0) 부여
    ↓
score < 0.7 → 해당 아티클 전체 스킵 (퀴즈 생성 안 함)
score ≥ 0.7 → 다음 단계 진행
```

### 개선 효과
- 할루시네이션이 포함된 요약이 DB에 저장되는 경로 **구조적으로 차단**
- `faithfulness_score` 분포를 런마다 기록 → 프롬프트 품질 모니터링 가능
- faithfulness 미달로 탈락한 아티클 수(`faithfulness_failures`)를 추적해 수집 품질 파악

## 문제 3 — 크로스런 URL 중복 (Cross-Run Deduplication)

### 문제
런 내부에서는 URL 중복을 제거했지만, **어제 수집한 아티클이 오늘 다시 수집되는 것을 막지 못했다.**  
같은 원문으로 퀴즈가 반복 생성되면:
- 유저가 동일 내용의 퀴즈를 반복해서 보게 됨
- DB에 중복 콘텐츠가 쌓여 저장·비용 낭비

### 해결 방법 — 수집 시점에 DB 기존 URL 배치 조회

```python
# 수집된 URL들을 한 번의 DB 쿼리로 확인
existing_urls = DB.contents.select("original_url").in_(urls)
raw_contents = [item for item in raw_contents
                if item["url"] not in existing_urls]
```

아티클 수집 직후, DB에 이미 저장된 URL을 일괄 제거한 뒤 요약 단계로 넘긴다.

### 개선 효과
- 기존 수집 아티클의 재처리를 **완전히 차단**
- `dedup_filtered` 수치로 중복 제거 건수 추적 가능
- 파이프라인이 오래 운영될수록 효율 향상 (기존 콘텐츠가 쌓일수록 중복 방지 효과 증가)

## 문제 4 — 아티클 관련성 판단 불가 (Context Relevance)

### 문제
Claude는 아티클의 `title`과 `text_length`만 보고 관련성을 판단했다.  
텍스트 길이는 품질과 무관하고, 제목만으로는 실제 내용이 토픽과 맞는지 알 수 없다.

### 해결 방법 — text_preview 300자 노출

수집 결과에 `text_preview` (원문 앞 300자) 를 추가로 포함해 Claude에 전달한다.  
Claude는 실제 내용의 일부를 직접 보고 관련성을 판단해 스킵 여부를 결정한다.

| 개선 전 Claude가 보는 것 | 개선 후 Claude가 보는 것 |
|------------------------|------------------------|
| title, source, text_length | title, source, text_length, **text_preview** |

### 개선 효과
- 관련 없는 아티클이 요약 단계까지 넘어가는 것을 앞단에서 차단
- 추가 API 호출 없이 Claude의 판단 품질만 향상 (비용 중립)

## 문제 5 — 원문 텍스트 절단 (Context Truncation)

### 문제
요약과 퀴즈 생성 모두 원문 앞 **3,000자만** 사용했다.  
긴 학술 논문의 경우 핵심 실험 결과나 결론이 뒷부분에 위치하는 경우가 많아,  
중요한 내용이 처리되지 못하는 구조적 손실이 있었다.

### 해결 방법 — 처리 텍스트 한도 2배 확대

| 항목 | 개선 전 | 개선 후 |
|------|---------|---------|
| 요약 입력 | `text[:3,000자]` | `text[:6,000자]` |
| 퀴즈 생성 입력 | `text[:3,000자]` | `text[:6,000자]` |

GPT-5의 넓은 컨텍스트 윈도우를 활용해 더 많은 원문을 처리한다.

### 개선 효과
- 원문 커버리지 **2배** 확대
- 논문 후반부 실험 결과·결론을 퀴즈에 반영 가능
- 요약의 완결성 향상

## 문제 6 — 커리큘럼 품질 미검증 (Curriculum Groundedness)

### 문제
유저가 새 관심사를 추가하면 Claude가 커리큘럼을 자동 생성한다.  
이 과정에서 **존재하지 않는 논문명·알고리즘명이 생성되어도 탐지할 방법이 없었다.**  
커리큘럼 구조(챕터 수, 필수 필드 등)도 검증 없이 DB에 저장됐다.

### 해결 방법 — 2단계 검증 도입

**1단계: 구조 검증 (코드)**
- 챕터 수 최소 6개 확인
- 필수 필드(`title`, `description`, `level`, `concepts`, `search_hints`) 존재 확인
- 위반 시 경고 로그 출력

**2단계: Concepts 실존 검증 (Claude 교차 검토)**
```
Claude (생성) → 커리큘럼 초안
    ↓
Claude (검토) → "이 concepts 목록이 실제로 존재하는가?"
    → 존재하지 않는 항목 수정 또는 제거
    → 챕터 순서 논리 검토
    ↓
검증된 커리큘럼 → DB 저장
```

또한 프롬프트 자체를 강화해 "막연한 표현 금지, 지어내지 말 것"을 명시적으로 지시했다.

### 개선 효과
- 존재하지 않는 개념이 학습 경로에 포함되는 것을 **이중으로 차단**
- 구조적 오류(필드 누락, 챕터 부족) 즉시 탐지
- 커리큘럼 생성은 최초 1회만 발생 → 추가 검증 비용 최소화

---

## 문제 7 — 품질 측정 수단 부재 (Quality Observability)

### 문제
파이프라인이 매일 실행되지만, **콘텐츠 품질이 얼마나 되는지 알 방법이 없었다.**  
성공/실패 로그는 있었지만 "저장된 콘텐츠가 좋은가"는 기록하지 않았다.

### 해결 방법 — 런마다 품질 지표 자동 집계

매 파이프라인 실행 완료 시 다음 지표를 측정하고 출력한다:

| 지표 | 의미 |
|------|------|
| `faithfulness_score` (평균) | 요약이 원문에 얼마나 충실한가 (0~1) |
| `faithfulness_failures` | faithfulness 미달로 스킵된 아티클 수 |
| `quiz_pass_rate` (평균) | 퀴즈 교차검증 통과율 |
| `dedup_filtered` | 크로스런 중복으로 제거된 아티클 수 |

```
[품질] faithfulness 평균=0.89 | faithfulness 탈락=2건 | 퀴즈 통과율 평균=78% | 크로스런 중복제거=5건
```

이 수치들은 `pipeline_runs.stats`에 저장되어 시계열로 추적 가능하다.

### 개선 효과
- 품질 저하를 **다음 날 확인이 아닌 당일 런에서 즉시 감지**
- 프롬프트 수정 전후 품질 변화를 수치로 비교 가능
- 이상 징후(faithfulness 급락, 통과율 급락) 조기 발견

## 전체 개선 구조 (개선 후)

```
[개선 후]

아티클 수집 (arxiv + Tavily)
    ↓  런 내 URL 중복 제거
    ↓  크로스런 URL 중복 제거 (DB 조회)  ← NEW
    ↓  text_preview 포함 → Claude 관련성 판단 ← NEW
GPT-5 → 요약 생성                        ← 모델 업그레이드
    ↓
Claude Haiku → faithfulness 검증 (score 0~1)   ← NEW (교차 검증)
    ↓  score < 0.7 → 스킵
GPT-5 → 퀴즈 생성                        ← 모델 업그레이드, 6000자
    ↓
Claude Haiku → 퀴즈 교차 검증            ← NEW (이종 모델)
    ↓  verified_count=0 → 스킵
DB 저장
    ↓
품질 지표 집계·출력                       ← NEW
```

## 한계 — 아직 완전하지 않은 것

- **커리큘럼 교차 검증이 동일 회사 모델**: 생성과 검증 모두 Claude Haiku 사용. 퀴즈와 달리 이종 모델 적용 어려움
- **Tavily 관련성 점수 의존**: 도메인 신뢰도와 토픽 관련성을 여전히 혼합해서 계산
- **유저 피드백 루프 없음**: 퀴즈 품질에 대한 유저 반응이 파이프라인 개선에 반영되지 않음
- **GPT-5 가격**: gpt-4o-mini 대비 약 100배 비용 증가. 처리량 많아질 경우 재검토 필요
